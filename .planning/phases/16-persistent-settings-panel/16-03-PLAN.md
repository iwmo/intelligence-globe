Boot wiring pattern:
```typescript
<GlobeView onViewerReady={(v) => {
  registerViewer(v);
  setCesiumViewer(v);
  // Apply settings defaults after viewer is live
  const s = useSettingsStore.getState();
  const appStore = useAppStore.getState();
  Object.entries(s.defaultLayers).forEach(([layer, visible]) => {
    appStore.setLayerVisible(layer as keyof typeof s.defaultLayers, visible);
  });
  appStore.setVisualPreset(s.defaultPreset);
  appStore.setReplayMode(s.defaultMode);
  if (s.defaultCamera) {
    flyToLandmark(s.defaultCamera);
  }
}} />
```

CRITICAL ordering constraint (from research pitfall #1):
- Layer and mode defaults: can apply at any point inside onViewerReady (useAppStore has no viewer dependency)
- Camera default: MUST be inside onViewerReady — flyToLandmark calls _viewer.camera.flyTo; before viewer is registered _viewer is null

Pitfall #5 (layer fetch/cancel on first load):
- useAppStore initializes layers with hardcoded defaults (satellites:true, aircraft:true)
- Layer components begin fetching immediately on mount, before onViewerReady fires
- Applying setLayerVisible in onViewerReady may trigger a redundant fetch+cancel for layers turned off
- This is acceptable — it is one network request and invisible to the user; do NOT try to apply layers earlier
