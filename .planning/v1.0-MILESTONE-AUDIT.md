## Cross-Phase Integration Findings

### Critical Gap: Alembic Migration Not Automated

**Affects:** INFRA-01, INFRA-02, SAT-01, AIR-01

On a clean `docker compose up`, the backend Dockerfile CMD runs uvicorn directly:
```
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

`app.main` lifespan calls `init_db()` which creates the PostGIS extension but **not** the `satellites` or `aircraft` tables. Those tables are created exclusively by Alembic migrations:
- `ac9bb4b6e929_add_satellites_table.py`
- `ca281e8bedd2_add_aircraft_table.py`
- `c5795b11a549_add_aircraft_latlon_index.py`

No entrypoint script chains `alembic upgrade head` before uvicorn starts. On a fresh deploy:
- Worker immediately fails on satellite/aircraft upsert (tables missing)
- `/api/satellites/` and `/api/aircraft/` return 500 until migrations are run manually

**Fix required:** Add `entrypoint.sh` to backend image:
```bash
#!/bin/sh
alembic upgrade head
exec uvicorn app.main:app --host 0.0.0.0 --port 8000
```
Or add a `command` override in `docker-compose.yml` backend service.

### Minor: SAT-03 Fly-To Silent Failure on Early Search

`SearchBar.tsx` posts `GET_POSITION` to `workerRef.current` without a null guard. The worker is created inside `SatelliteLayer` only after `satellites.data` arrives (up to 30 seconds). If a user searches before that fetch completes, the camera fly-to silently does nothing. The satellite detail panel still opens correctly via `setSelectedSatelliteId`. UX gap only — not a data loss or crash.

### Negligible: Dead Store Slice

`searchQuery` / `setSearchQuery` defined in `useAppStore.ts` but never read or written outside the store. `SearchBar.tsx` uses local `useState` for its query string. Zero runtime impact.
